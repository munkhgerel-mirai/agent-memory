# Release Notes: AI-DLC Template Workflow Hardening

**Version:** v0.1.0  

## Overview

This release consolidates the changes that harden the AI-DLC template into a skill-based, approval-gated, and traceable workflow.

The main goal is to ensure that agents and humans document each project lifecycle step clearly: what was planned, who approved it, what risks exist, and which verification evidence supports the result.

## Highlights

- The active workflow is centered on explicit `ai-dlc-*` skills.
- Plan-first execution and the human approval gate are now clearer across all lifecycle steps.
- Approved plans are treated as historical decision records.
- Technology choices do not become binding until they pass the Human Selection Gate and ADR approval.
- Lifecycle templates have moved beyond empty document shells and now act as readiness controls for downstream phases.
- The quality gate and skill-to-artifact map make each artifact's inputs, outputs, approvals, and handoffs explicit.

## Changed

- Inception templates now capture intent, stakeholders, measurable outcomes, scope, edge cases, assumptions, open questions, and NFRs more completely.
- Unit and Bolt templates now make cohesive boundaries, story/NFR/risk traceability, dependencies, validation methods, and exit criteria more explicit.
- Domain Design now focuses on technology-agnostic DDD artifacts so the domain model is not polluted by database, framework, or cloud choices.
- Design Plan now records inputs, approval status, plan integrity, clarifications, execution notes, and verification results more explicitly.
- Code Generation, Deployment, and Observability templates now require traceability and validation evidence from implementation through production readiness.
- The quality gate checklist now brings together review surfaces for universal gates, setup validation, inception, stories, NFRs, risks, units, bolts, discovery, design, technology decisions, code generation, deployment, and operations.
- The skill-to-artifact map now lets agents follow approved artifact handoffs instead of guessing the next step.

## Governance And Constraints

- The plan-first workflow is preserved: write the plan first, get human approval, and do not begin implementation before approval.
- The human approval gate must not be skipped during acceleration, automation, or refactoring work.
- Reusable templates and methodology references must not be accidentally damaged during project-specific cleanup, placeholder replacement, or workflow updates.
- Technology mapping inside Logical Design is a non-binding recommendation, not implementation approval.
- A technology decision becomes binding only after human selection, approval, and downstream authorization are recorded.
- An approved plan is not a current TODO list; it is a record of the scope, rationale, and decisions approved at that time.

## Rejected Approaches

- Treating Logical Design technology mapping as implementation approval was rejected because it could lead to implementing AI-suggested options before a human approves them.
- Semantically rewriting an approved plan to fit later circumstances was rejected because it would erase the decision history of what was approved at the time.
- Letting each agent infer artifact readiness independently was rejected because a shared quality gate and skill-to-artifact map provide a more verifiable process.

## Directives For Future Work

- Do not make AI-suggested technology choices such as stacks, frameworks, databases, cloud services, or libraries binding without human selection, approval, and downstream authorization.
- Do not rewrite the meaning, scope, rationale, or decisions of an approved plan after approval.
- During plan execution, only update checkbox status or add execution notes, blockers, verification results, or approved addenda.
- Runtime code and tests belong in approved runtime and test folders.
- Before downstream work starts, verify that the required upstream artifact has been approved.
