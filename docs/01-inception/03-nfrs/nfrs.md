# Non-Functional Requirements

**Project:** Agent-memory

## Approval Status

Pending human review. Generated from the approved Inception plan and approved Mob Elaboration answers.

| ID | Category | Requirement | Target Measure | Rationale | Verification Method | Applicability |
|----|----------|-------------|----------------|-----------|---------------------|---------------|
| NFR-001 | Performance | Startup context retrieval must be fast enough for fresh agent sessions. | Return current goal, phase, active plan, approved decisions, blockers, and next steps within 10 seconds for a local workspace at v1 scale. | Fresh-session continuity is the primary product outcome. | Automated retrieval test with representative AI-DLC artifacts. | Applicable |
| NFR-002 | Performance | Default startup context pack must stay compact. | Context pack must not exceed 2000 tokens unless an explicit expanded retrieval mode is requested. | Controls token spend while preserving enough project rationale. | Context pack token-count check in tests. | Applicable |
| NFR-003 | Reliability | SQLite index must be rebuildable from durable sources. | Rebuild succeeds from Markdown artifacts and JSONL event log after deleting the SQLite index. | SQLite is an index/cache, not the sole source of truth. | Rebuild integration test. | Applicable |
| NFR-004 | Reliability | Historical approved artifacts must remain stable records. | Later state changes must not rewrite prior approved plans; new state is recorded in current artifacts or addenda. | AI-DLC traceability depends on decision-record integrity. | Document workflow review and artifact diff checks. | Applicable |
| NFR-005 | Usability | Solo developers must be able to run v1 locally. | Local setup works without hosted infrastructure for MCP, CLI, local API, Markdown docs, JSONL, and SQLite. | Low-friction local adoption is required before team/server expansion. | Setup smoke test and README walkthrough. | Applicable |
| NFR-006 | Usability | Retrieved context must be explainable to humans and agents. | Each durable memory included in context includes source path/provenance and memory category. | Prevents opaque memory injection and supports review. | Context pack snapshot review. | Applicable |
| NFR-007 | Security | Secrets must not be stored automatically in durable memory. | Secret-like values are redacted or blocked before durable indexing. | Memory is long-lived, so accidental secret persistence is high risk. | Secret redaction tests and sample scan. | Applicable |
| NFR-008 | Security | Durable memory writes must respect approval and visibility policy. | Durable shared memory includes approval status and visibility scope. | Prevents draft/private context from being treated as team truth. | Unit tests for policy evaluation. | Applicable |
| NFR-009 | Scalability | v1 must handle practical local workspace memory. | Support at least 1,000 lifecycle memory records and 10,000 raw/event records in a local workspace without exceeding the 10-second startup target. | Gives enough room for real AI-DLC project use without premature server complexity. | Local performance test with generated representative data. | Applicable |
| NFR-010 | Scalability | Architecture must allow future server profile migration. | Storage interfaces separate domain model from SQLite implementation and allow future Postgres + pgvector profile. | Team/server use may later require higher concurrency and access control. | Architecture review and interface tests. | Applicable |
| NFR-011 | Compliance | Memory must support deletion and export from day one. | Durable memory can be exported; deleted memory is removed from active retrieval indexes, lifecycle edges, and optional vector entries when present. | Supports user control, auditability, and future compliance needs. | Export/delete integration tests. | Applicable |
| NFR-012 | Compliance | Enterprise compliance controls are deferred but not blocked. | Domain metadata includes actor, source, timestamp, visibility, retention, and approval status. | Future enterprise use needs evidence and control hooks. | Metadata schema review. | Applicable |
| NFR-013 | Operability | Operators need inspect, rebuild, export, delete, and query commands. | CLI exposes basic workspace memory operations for v1. | Local-first systems still need transparent operations. | CLI smoke tests. | Applicable |
| NFR-014 | Operability | Runtime jobs should be observable when iii-engine is enabled. | iii-engine adapter jobs record job result, timing, and linked provenance. | Optional runtime orchestration should improve operations, not hide them. | Adapter integration test or manual runtime trace. | Applicable when iii-engine adapter is enabled; otherwise deferred. |
| NFR-015 | Maintainability | Core domain model must stay independent from runtime and storage implementations. | Domain concepts do not import iii-engine, SQLite, Postgres, or embedding provider details directly. | Keeps Agent-memory portable and avoids early lock-in. | Architecture/code review. | Applicable |
| NFR-016 | Maintainability | AI-DLC memory categories must be explicit and testable. | Memory categories map to lifecycle artifact types with validation coverage. | The product differentiator is lifecycle-aware memory, not generic notes. | Unit tests for classification. | Applicable |
| NFR-017 | Privacy | Raw observations should have shorter retention than approved artifact memories. | Default raw observation TTL is defined before implementation; approved lifecycle memories retain longer. | Limits accumulation of unnecessary sensitive raw context. | Policy review and retention test. | Applicable; exact TTL open. |
| NFR-018 | Privacy | PII and sensitive workspace data require policy classification before durable storage. | Sensitive categories are blocked, redacted, or require explicit approval according to policy. | Memory should be safe by default. | Policy tests and redaction review. | Applicable |
| NFR-019 | Cost | v1 should not require paid hosted infrastructure by default. | Local Markdown, JSONL, SQLite, MCP, CLI, and local API work without managed services. | Keeps adoption cheap and avoids early ops burden. | Setup verification. | Applicable |
| NFR-020 | Cost | Token use must be bounded by retrieval mode. | Default startup mode <= 2000 tokens; expanded modes require explicit request. | Prevents runaway context injection cost. | Token budget tests. | Applicable |
